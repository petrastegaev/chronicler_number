
# Game Design Document: «Дуэль чисел» (расширенная версия)

## 1. Общая информация
- **Название:** Дуэль чисел
- **Жанр:** Локальная дуэльная викторина с числовыми ответами
- **Платформа:** Веб-приложение, работающее полностью офлайн в локальной сети
- **Режим игры:** 1 на 1, 9 раундов
- **Устройства:** 
  - Сервер (ноутбук) с запущенным бэкендом и базой данных
  - Два клиентских устройства для игроков (ноутбук/планшет)
  - Устройство администратора (телефон/планшет/ноутбук) с админ-панелью
- **Стек:** FastAPI (Python), React (JavaScript), WebSocket, PostgreSQL
- **Сеть:** Офлайн, локальная Wi‑Fi сеть

## 2. Концепция игры
Два игрока сидят напротив друг друга, каждый видит свой экран. Перед началом игры они вводят свои ники. В каждом раунде появляется одинаковый вопрос с числовым ответом. За 10 секунд игроки вводят целое число. Побеждает тот, кто ближе к правильному ответу по модулю разности. После 9 раундов определяется победитель по сумме очков. Игра сопровождается звуковыми эффектами: тиканье таймера, сигнал окончания раунда, фанфары при объявлении итогов.

## 3. Игровой процесс

### 3.1. Подготовка к игре
1. Игроки открывают в браузере единый адрес (например, `http://<server-ip>:8000/`).
2. На экране отображается поле ввода ника и кнопка «Готов».
3. Первый введший ник становится «Игрок 1», второй — «Игрок 2» (автоматическое распределение сервером). На админ-панели появляются ники подключённых игроков.
4. Когда оба игрока готовы (подтвердили ники), админ может запустить игру.

### 3.2. Структура раунда (один из 9)
Сохраняется как в исходном ТЗ, дополнено звуками:
- При старте раунда начинается 10-секундный обратный отсчёт с тикающим звуком (раз в секунду).
- В последние 3 секунды звук тика становится чаще или меняется тональность.
- По истечении времени раздаётся гонг/звук завершения, ответы фиксируются.
- Во время паузы 2–3 секунды проигрывается короткий джингл.
- При объявлении итогов игры звучат фанфары.

### 3.3. Начисление очков и победитель – без изменений.

## 4. Вопросы
- Пул вопросов минимум 60.
- Добавление вопросов осуществляется **через админ-панель**:
  - **Ручное добавление:** форма с полями «Текст вопроса», «Правильный ответ» (целое число), «Категория» (опционально).
  - **Массовая загрузка из CSV:** админ может выбрать файл формата `text,answer[,category]`. Сервер парсит файл и добавляет все валидные строки в пул.
- Используются вопросы без единиц измерения и текста в ответе.
- В игре 9 случайных вопросов без повторов (либо предустановленный сет, если реализовано).

## 5. Интерфейс игрока

### 5.1. Единая точка входа
URL: `/` или `/play`. Первые экраны:
- **Экран ввода ника:** текстовое поле (макс. длина 15 символов), кнопка «Присоединиться». После подтверждения ник отправляется на сервер, и экран переходит в состояние ожидания начала игры.
- Если оба игрока ещё не готовы, отображается сообщение «Ожидание соперника…».
- Когда оба готовы, появляется надпись «Ожидание запуска администратором».

### 5.2. Основной игровой экран (после старта)
- В верхней части: номер раунда (например, «Раунд 3 / 9») и текущий счёт игрока.
- По центру крупно: текст вопроса.
- Поле ввода ответа (только целые числа, ограничение визуальное и серверное 0–1 000 000).
- Крупный таймер обратного отсчёта.
- Во время показа результата раунда:
  - Правильный ответ, ответ игрока, ответ соперника.
  - Цветовое выделение победителя раунда.
- **Звуковое сопровождение** (см. раздел 8).

### 5.3. Итоговый экран
- Финальный счёт, ник победителя (или «Ничья»).
- Проигрываются фанфары.
- Кнопка «Играть снова» не предусмотрена (управление через админа).

## 6. Админ-панель (расширенная)

### 6.1. Управление игрой
- Запуск игры (доступен только когда оба игрока ввели ники и отмечены как готовые).
- Быстрый рестарт (сбрасывает игру в исходное состояние готовности).
- Отображение ников подключённых игроков, их статус (онлайн/нет).
- Текущий раунд и счёт по каждому игроку.

### 6.2. Управление вопросами
Новый раздел «Вопросы» с двумя подрежимами:
1. **Добавить один вопрос**
   - Форма: текст вопроса, числовой ответ (обязательно), категория (опционально).
   - Валидация на стороне клиента и сервера.
2. **Импорт из CSV**
   - Кнопка выбора файла `.csv`.
   - Предварительный просмотр первых 5 записей.
   - Подтверждение загрузки. Сервер обрабатывает записи, пропуская строки с ошибками (нечисловой ответ, пустой текст), возвращает количество успешно добавленных вопросов и список ошибок.
3. **Просмотр всех вопросов** (таблица с пагинацией) с возможностью удаления отдельного вопроса (опционально).

### 6.3. Статистика
- Кнопка «Статистика» показывает общее количество проведённых игр (из БД).

## 7. Архитектура системы (дополнения)

### 7.1. Клиентские роли и WebSocket
- Игроки подключаются через WebSocket к общему эндпоинту `/ws`.
- При соединении они отправляют сообщение `join` с данными:
  ```json
  {
    "event": "join",
    "data": {
      "role": "player",   // или "admin"
      "nickname": "Алекс" // только для игроков
    }
  }
  ```
- Сервер назначает роли player1 и player2 в порядке подключения. Ник сохраняется в сессии.
- Если игрок переподключается (разрыв связи), сервер восстанавливает его роль и ник по идентификатору сессии (куки/токен не обязателен, можно по IP, но надёжнее использовать session id из cookie, устанавливаемой при первом HTTP-запросе).

### 7.2. REST API для управления вопросами
- `POST /api/questions` – создать один вопрос (тело JSON: `{text, answer, category?}`).
- `POST /api/questions/upload-csv` – загрузка CSV-файла (multipart/form-data). Возвращает `{added: N, errors: [...]}`.
- `GET /api/questions` – список всех вопросов (для админ-панели).
- `DELETE /api/questions/{id}` – удалить вопрос.

Админ-панель использует эти эндпоинты напрямую (через HTTP), а не WebSocket.

### 7.3. База данных
Схема остаётся прежней (таблицы `questions`, `game_sessions`, `rounds`, `stats`). Добавляется поле `nickname` в сессию игроков, но так как статистика по никам не обязательна, можно хранить ники только в оперативной памяти (в объекте игровой сессии). Если требуется сохранение, можно добавить таблицу `players` или поля `player1_nick`, `player2_nick` в `game_sessions`.

### 7.4. Синхронизация таймера и звуков
- Таймер управляется сервером: каждую секунду рассылается событие `timer_tick` с оставшимся временем.
- Клиенты воспроизводят звук тика при получении события `timer_tick`. При оставшемся времени ≤ 3 секунд клиент переключает звуковой эффект на ускоренный/повышенный тон.
- Событие `round_result` и `game_end` содержат указание воспроизвести финальные звуки. Клиент самостоятельно воспроизводит аудиофайлы (локально загруженные в сборку).

## 8. Звуковые эффекты

### 8.1. Ресурсы
Аудиофайлы в формате `.mp3` или `.ogg`, включены в production-сборку React (папка `public/sounds`):
- `tick.mp3` – короткий щелчок для обычного отсчёта (0.1 сек).
- `tick_fast.mp3` – для последних 3 секунд.
- `end_round.mp3` – гонг/звонок окончания раунда.
- `winner.mp3` – фанфары для окончания игры.

### 8.2. Логика воспроизведения
- Используется HTML5 Audio API или библиотека наподобие Howler.js для гарантированного воспроизведения на мобильных устройствах без автозапуска проблем (после первого взаимодействия пользователя – нажатия кнопки «Готов» активируется AudioContext).
- Клиент реагирует на события WebSocket:
  - `timer_tick` → `play('tick')`, если `remaining > 3`; иначе `play('tick_fast')`.
  - `round_result` → `play('end_round')`.
  - `game_end` → `play('winner')`.

## 9. Технические требования (дополнения)

### 9.1. Локальная статика
Все файлы (React build, звуки, шрифты) раздаются FastAPI через `StaticFiles`. Никаких внешних CDN.

### 9.2. Поддержка CSV на бэкенде
Установить библиотеку `python-multipart` для обработки загрузки файлов и стандартный модуль `csv` для парсинга. Валидация: проверка, что `answer` – целое число в диапазоне 0..1000000, текст не пустой.

### 9.3. Ники и управление состоянием игроков
Сервер хранит активные WebSocket-соединения с метаданными: `role`, `nickname`, `connected`. При старте игры ники фиксируются в объекте `GameSession`. При показе результатов они используются для отображения на экранах и в админке.

## 10. Статистика (изменения)
Помимо счётчика игр, теперь можно хранить ники победителя и участников, что позволяет показывать простую статистику «Побед/поражений/ничьих» для каждого ника (опционально). В минимальном варианте статистика остаётся счётчиком игр.

## 11. Обновлённые макеты пользовательского интерфейса (кратко)
- **Экран входа:** по центру «Введите ваш ник», поле и кнопка «Готов». При нажатии кнопка меняется на «Ожидание…».
- **Админ-панель:** вкладки «Игра», «Вопросы». Вкладка «Вопросы» содержит подвкладки «Список», «Добавить», «Импорт CSV».

## 12. Заключение
Все дополнительные функции интегрированы в единую архитектуру, не нарушая офлайн-работу. Именование игроков делает игру более персонализированной, звуки повышают вовлечённость, а управление вопросами через админ-панель и CSV-загрузку упрощает администрирование без необходимости редактировать базу данных вручную. Разработка может вестись поэтапно, опираясь на этот документ.
