## Interaction Contract

### Screen State Machine

Single `<GameScreen>` component at route `/` renders differently based on Zustand `phase` field:

```
idle --> joining --> waiting --> playing --> showing_result --> playing (loop)
                                                                    |
                                                                    v
                                                               finished
```

| Phase | Sub-component | Description | Transitions |
|-------|---------------|-------------|-------------|
| `idle` | `<JoinScreen>` | Nickname input + join button. Initial state. | -> `joining` on join click |
| `joining` | `<JoinScreen>` | Nickname sent, waiting for server `joined` confirmation. Button disabled, shows "Отправка...". | -> `waiting` on confirmation |
| `waiting` | `<WaitingScreen>` | Connected, waiting for game start. Shows "Ожидание соперника..." if alone, "Ожидание запуска администратором" if both ready. | -> `playing` on `game_started` event |
| `playing` | `<PlayingScreen>` | Question, timer, answer input, submit button. Active round. | -> `showing_result` on `round_result` event |
| `showing_result` | `<ResultOverlay>` | Full-screen overlay with round outcome. | -> `playing` on `round_started` event (next round), or -> `finished` on `game_end` event |
| `finished` | `<FinalScreen>` | Final scores, winner/draw declaration, waiting for admin restart. | -> `waiting` on `game_reset` event |

### WebSocket Connection Lifecycle

1. **Connect:** On page load, open WebSocket to `ws://{host}:8000/ws` (same host as page, derived from `window.location`). Include credentials for session cookie.
2. **Join:** After nickname submitted, send `join` with role `"player"` and nickname. Store WebSocket reference in Zustand store.
3. **Receive events:** Event listener dispatches to store actions:
   - `joined` -> set `playerNumber`, move to `waiting`
   - `game_started` -> set nicknames, move to `playing`, set `currentRound = 0`
   - `round_started` -> set question text, round number, reset `remaining` to 10, move to `playing` (if coming from `showing_result`)
   - `timer_tick` -> set `remaining`
   - `round_result` -> store result data, move to `showing_result`
   - `game_end` -> store final data, move to `finished`
   - `game_reset` -> clear all state, move to `waiting`
   - `state_snapshot` -> restore state on reconnect
4. **Send:** On early submit, send `submit_answer` with answer value. On auto-submit at timer expiry, same action.
5. **Disconnect/reconnect:** On disconnect, show connection error overlay. Reconnect WebSocket; server sends `state_snapshot` on successful re-auth via session cookie.

### Timer Visualization (D-01, D-02, D-03)

- SVG circle ring, 88px outer diameter, 6px stroke width, counter-clockwise stroke-dashoffset animation.
- Digits displayed inside the ring: 64px bold Inter, centered.
- Motion library `animate` controls the stroke-dashoffset transition (300ms ease-out per tick, synced to 1-second server interval).
- Color defaults to player's own accent (`player1` blue or `player2` red).
- Color transitions: `player1`/`player2` -> `warning` (#F59E0B) at remaining <= 5 -> `danger` (#EF4444) at remaining <= 3.
- No scale animation on digits at low time (Phase 5 audio tick provides the urgency cue).

### Answer Input (D-08, D-09, D-10, D-11)

- `<input type="number" min="0" max="1000000">`, full-width, 36px font, minimum 56px height.
- Triggers numeric keyboard on tablets (type="number").
- Immediate local echo — value updates on keystroke, no server round-trip.
- Submit button («Ответить») adjacent to input: primary action button, 56px height, rounded, white text on player accent color background.
- On click: disable input and button, show «Ответ принят» confirmation text replacing the button, send `submit_answer` via WebSocket.
- On timer expiry without early submit: auto-submit current input value. If empty or invalid (NaN, < 0, > 1000000): send null (treated as no answer server-side).

### Round Result Overlay (D-04, D-05, D-06, D-07)

- Full-screen overlay with semi-transparent dark background (`rgba(26, 10, 46, 0.85)`).
- Centered card (`wb-surface` background, 24px padding, rounded-xl) fades in (Motion fade+scaleY, 300ms ease-out).
- Display duration: ~3 seconds (server-controlled via `asyncio.sleep(3)` between `round_result` and next `round_started`).
- Layout (vertical stack, centered text):
  1. **Winner indicator:** «Вы выиграли раунд!» or «Соперник выиграл раунд» or «Ничья» — 24px bold, player accent color for win, danger for loss, white for draw.
  2. **Your answer:** «Ваш ответ: {value}» — 28px semi-bold, player accent color if winning, white otherwise.
  3. **Correct answer:** «Правильный ответ: {value}» — 28px semi-bold, `correct` green (#10B981).
  4. **Opponent's answer:** «Ответ соперника: {value}» — 24px semi-bold, white.
- Auto-dismisses when next `round_started` event received. Smooth fade-out 200ms.

### Header Bar (D-14)

- Present on all screens during `playing`, `showing_result`, and `finished` phases.
- Positioned at top, full-width, `wb-surface` background, 8px vertical padding, 16px horizontal padding.
- Left: player's own nickname (16px, player accent color).
- Center: game title «Дуэль чисел» (14px, muted text).
- Right: score (20px bold, player accent color) + round indicator (14px, muted text: «Раунд N / 9»).
- Not present on `idle`/`joining`/`waiting` screens — those screens show the title as a centered display element instead.

### Crossfade Transitions (D-12, D-13)

- Motion `AnimatePresence` wraps the active screen sub-component.
- Transition: fade duration 300ms, slight vertical slide (20px) for entry/exit.
- Result overlay enters with 300ms fade+scale, exits with 200ms fade.

## Component Inventory

Components to be built for Phase 3 (within the player frontend scope):

| Component | Purpose | States | Responsive Notes |
|-----------|---------|--------|-----------------|
| `<GameScreen>` | Root component at route `/`, renders sub-component based on phase | idle, joining, waiting, playing, showing_result, finished | Full viewport height, centered layout |
| `<JoinScreen>` | Nickname input + join button | idle, submitting | Centered vertically, max-width 400px card |
| `<WaitingScreen>` | Waiting states for opponent / admin start | waiting_opponent, waiting_admin, both_ready | Centered text with animated dots |
| `<PlayingScreen>` | Active round: question, input, timer, header | playing, answered | Full-width, question section ~40% height, input section ~30% height, timer positioned upper-right of center |
| `<TimerRing>` | SVG circular countdown | 10+ seconds, <=5s (warning), <=3s (danger), expired | 88px diameter, absolute positioned |
| `<AnswerInput>` | Numeric input + submit button | idle, answered (disabled), auto-submit | Full-width, 56px min-height |
| `<ResultOverlay>` | Full-screen overlay with round outcome | enter (animate in), display, exit (animate out) | Full viewport, centered card |
| `<FinalScreen>` | Game end summary | initial (animate in) | Centered vertical stack |
| `<GameHeader>` | Top bar with nickname, title, score, round | visible during playing/showing_result/finished | Full-width bar |
| `<ConnectionStatus>` | WebSocket connection indicator / error | connected, disconnected | Full-screen overlay on disconnect |

All components must be keyboard-accessible. Input fields must have associated labels. Timer must have an `aria-live` region announcing remaining seconds.
