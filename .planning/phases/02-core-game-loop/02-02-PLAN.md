<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| WebSocket client -> Server | Player or admin sends JSON messages. The server must validate every event and enforce role-based routing. |
| Player client -> opponent | Players are on separate devices; answers must never leak mid-round. |
| Reconnecting client -> Server | Client passes session token as query param. Server validates and restores role/slot. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Spoofing | WebSocket event dispatch (main.py) | mitigate | Role-based routing in event dispatch loop: `submit_answer` only accepted from `manager.player1` or `manager.player2` WebSocket references; `start_game`/`restart` only accepted from `manager.admin`. A player WebSocket connection cannot inject admin events — the WebSocket object reference is checked, not the message content. |
| T-02-02 | Tampering | GameSession.submit_answer() | mitigate | First-answer-wins policy: subsequent submissions from the same player in the same round are silently ignored (D-12). Duplicate or replayed WebSocket messages cannot double-submit. |
| T-02-03 | Information Disclosure | Answer secrecy mid-round | mitigate | GameSession NEVER broadcasts individual answers. Both answers are stored in instance variables and only revealed in the `round_result` event after timer expiry (GAME-04, D-13). No per-answer send/broadcast call exists in the GameSession code. |
| T-02-04 | Denial of Service | Timer manipulation | mitigate | Server-authoritative timer via `asyncio.sleep(1)` in GameSession (D-01, D-05). Client cannot control, speed up, or slow down the timer. Client clock manipulation has no effect on game timing. |
| T-02-05 | Tampering | Late answer after timer expiry | mitigate | `submit_answer()` checks `self.state == "accepting_answers"` before storing any answer. After the timer loop completes, state transitions to `showing_result` and all further submissions are silently ignored per D-12. |
| T-02-06 | Spoofing | Session token guessing (JOIN-05) | accept | UUID v4 hex tokens (32 chars, 122 bits of entropy). For a local WiFi booth with <100 reconnect attempts, collision odds are negligible. In-memory storage means tokens are never persisted or logged. Acceptable risk for offline booth demo. |
| T-02-07 | Tampering | GameSession persistence at game end | mitigate | SQLite writes are wrapped in `asyncio.shield()` (Pitfall 3) to survive task cancellation from WebSocket disconnect. Persistence is a single transaction: GameSession record + 9 Round records + Stat update, committed atomically. |
| T-02-08 | Tampering | Admin restart double-tap | mitigate | `reset_game()` guards against double-restart: checks `active_session is None` or `state == "waiting_for_players"` before proceeding. Idempotent — second restart call is ignored. |
| T-02-09 | Spoofing | Reconnect with invalid/expired token | mitigate | `restore_from_token()` returns None for unknown tokens. The reconnect path sends an error event and closes the WebSocket when token is invalid. |
| T-02-10 | Tampering | Double-session race on start_game | mitigate | `start_game` guard checks `game_task is not None and not game_task.done()` before creating a new GameSession. If a game is already running or starting, the event is silently ignored. This prevents the race where a second `start_game` arrives before the first GameSession.run() has transitioned out of `waiting_for_players` state. |

</threat_model>
