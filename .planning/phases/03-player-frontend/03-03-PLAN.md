# Phase 3 Plan 3: Gap Closure — Player Join Broadcast

**One-liner:** Fix the WaitingScreen "Ожидание запуска администратором" never shown by broadcasting opponent nickname on player join.

## Root Cause

From UAT Test 3:
- Server sends `joined` event ONLY to the connecting player, with no data about other connected players
- `player2Nickname` in frontend store is ONLY set by `setGameStarted()` which fires on `game_started` event from admin
- WaitingScreen checks `player2Nickname` — always empty until admin starts game

## Fix Strategy (Minimal)

### Server (backend/main.py)

In the player join handler (lines 157-172), two changes:

1. **Include opponent nickname in `joined` response** to the connecting player:
   - If joining as player 1: opponent = manager.player2_nickname (null if player 2 not yet joined)
   - If joining as player 2: opponent = manager.player1_nickname (always set since player 1 joined first)

2. **When player 2 joins, notify player 1** via `manager.send_to_player(1, ...)` with `player_joined` event containing player 2's nickname

### Frontend (frontend/src/stores/gameStore.ts + frontend/src/hooks/useWebSocket.ts)

1. **Add `setOpponentNickname` action** to gameStore that sets the opponent's nickname
2. **Handle `opponent_nickname` in `joined` event handler** — if present, set it via the new action
3. **Handle `player_joined` event** — when received, set `player2Nickname` from event data

## Tasks

### Task 1: Server join handler — include opponent nickname
- File: `backend/main.py`
- Modify the `joined` response at lines 165-172 to include `player1_nickname` and `player2_nickname` from manager
- After assigning player 2, broadcast `player_joined` to player 1

### Task 2: Frontend store + WebSocket handler
- File: `frontend/src/stores/gameStore.ts` — add `setOpponentNickname` action
- File: `frontend/src/hooks/useWebSocket.ts` — handle `opponent_nickname` in `joined` event and add `player_joined` event handler
